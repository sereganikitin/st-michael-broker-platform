# Логика уникальности клиента — текущее состояние (2026-05-14)

## Кратко

**Уникальность** — это право брокера вести конкретного клиента эксклюзивно. Если другой брокер пытается «привести» того же клиента, наша система это разрешает (multi-broker), но в кабинете каждого видна только его запись + статус.

## Период уникальности

- **30 дней** от даты создания первого лида клиента в amoCRM (поле `Client.amoCreatedAt`)
- Раньше было 30 → 40 → 30 (последняя правка пользователя)
- Истечение: `Client.uniquenessExpiresAt = amoCreatedAt + 30 дней`
- В кабинете под датой создания бейдж:
  - `осталось N дней (30 дн)` — обычный
  - `⚠ осталось ≤ 7 дн` — жёлтый
  - `⚠ уникальность истекла N дн назад` — красный

## Источник даты создания (`amoCreatedAt`)

Берётся **минимальная** `lead.created_at` среди всех связанных лидов amoCRM для этого клиента.

Пример: Ольга +79139882472 имеет 2 лида в amoCRM:
- `32005847` от **01.02.2026** (КЦ)
- `32077053` от **14.03.2026** (КЦ)

В нашей БД `Client.amoCreatedAt = 01.02.2026` (MIN). Уникальность истекла 03.03.2026 (= MIN + 30 дн).

## Multi-broker — каждый брокер ведёт своего клиента

Если разные брокеры заводят одного и того же клиента (по телефону):

- В amoCRM создаются **разные лиды** (по одному на каждого брокера, с разным `responsible_user_id`)
- В нашей БД создаются **отдельные `Client` записи** — по одной на каждого брокера (`brokerId` разный, `phone` одинаковый)
- Каждый брокер в своём кабинете видит **только свою** запись Анатолия со статусом `Уникален`

То есть оба брокера до тех пор пока не доказано обратное считают что Анатолий — их клиент.

**Сценарий**:
1. **13.05** Рэдкет фиксирует Анатолия → `Client(brokerId=Рэдкет, status=CONDITIONALLY_UNIQUE, expires=12.06)`
2. **14.05** Trend Agent фиксирует Анатолия → `Client(brokerId=Trend, status=CONDITIONALLY_UNIQUE, expires=13.06)`
3. У Рэдкета в кабинете: «Анатолий — Уникален»
4. У Trend Agent в кабинете: «Анатолий — Уникален»

## Фиксация из кабинета брокера (Фиксация)

Форма требует:
- Телефон клиента (10 цифр для RU или произвольный для иностранного — есть тоггл «Иностранный номер»)
- Имя (обязательно)
- Фамилия (необязательно)
- Проект (Зорге 9 / Серебряный бор)
- Тип (Квартира / Апартаменты / Коммерческая)
- Кол-во комнат (необязательно — Студия / 1 / 2 / 3 / 4+)
- Метраж м² (обязательно)
- Бюджет покупки ₽ (обязательно, с форматированием тысяч)

При отправке:
1. Создаётся `Client` в нашей БД с `brokerId=текущийБрокер`
2. Создаётся лид в amoCRM с custom-полями:
   - `587387` Тип объекта = propertyType
   - `583447` Сколько комнат рассматривает = roomsCount
   - `833045` Стоимость без скидок = бюджет
   - `604555` Метраж = sqm

## Синхронизация из amoCRM

Раз в 30 минут (`@Cron('*/30 * * * *')`) `SchedulerService.handleAmoCrmSync()` бежит по всем ACTIVE брокерам с `amoContactId`:

1. Берёт все лиды, привязанные к контакту брокера в amoCRM
2. Для каждого лида:
   - Если `pipeline_id === 10787390` (Воронка брокеров) → пропускает
   - Если `phone === '+70000<leadId>'` (fake fallback) → пропускает + удаляет старую запись
   - Если `pipeline_id === 7600542` (КЦ-pipeline) → создаёт/обновляет Client, но **не** создаёт Deal (это карточки встреч КЦ, не сделки)
   - Если `status_id === 143` («Закрыто и не реализовано») → удаляет существующий Deal из БД
   - Если статус **не** на сделочной стадии → удаляет существующий Deal
   - Иначе → создаёт/обновляет Deal с реальными данными из лида

## Что меняется при апдейте `Client` через sync

- `amoCreatedAt` → MIN из всех обработанных лидов клиента (никогда не позже)
- `amoUpdatedAt` → MAX
- `uniquenessExpiresAt` → пересчитывается = новый `amoCreatedAt` + 30 дней (когда MIN сдвигается на более раннюю дату)
- `fullName` → берётся из `contact.name` в amoCRM, очищается от суффиксов «от <X>» (например, «Лина от брокера» → «Лина»)

## Cleanup при синке

Перед обработкой лидов брокера sync удаляет «устаревшие» записи:
- `Meeting` + `Deal` + `Client` с phone `LIKE '+70000%'` (fake fallback номера)

В цикле:
- Если лид cancelled (143) или КЦ или не deal-stage → удаляет существующий Deal под этим `amoDealId`
- При найденном parent/child через `cc_id_parent` → дедуп: оставляет запись с большим `sqm`

## Что НЕ реализовано (отложено)

1. **Downgrade при попадании клиента в офис** — когда один из брокеров доводит клиента до сделочной стадии (`Deal.status = SIGNED`), у других брокеров уникальность должна автоматически сменяться на «Уникальность не подтвердилась» + аларм менеджеру/КЦ.
2. **Передача клиента менеджером** — есть **backend API** `PATCH /admin/clients/:id/reassign-broker` (вызывается админом/менеджером), но **UI кнопки нет**. Решено: новый брокер просто получает свою запись через обычный sync, без явной «передачи».
3. **Лендинг — шкала ставок** — в кабинете «Шкала ставок» скрыта при FLAT-политике, но на лендинге (главной странице) аналогичный фикс не сделан.
