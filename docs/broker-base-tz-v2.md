## 2. Рекомендации

### 2.1 Архитектура: расширить `Broker` (не создавать отдельный `BrokerLead`)

**Почему**: единая запись брокера от первого холодного звонка до закрытия сделки. Никакого «двойного учёта». Если брокер сам зарегистрируется — связь автоматическая.

**Минус**: таблица `brokers` разрастётся до 10K+ строк, но это норма для Postgres.

### 2.2 Категории — на основе текущих 11 результатов

Свести 11 категорий в **5 рабочих статусов**:

| Статус (`BrokerCategory`) | Что входит | Действие КЦ |
|---|---|---|
| `COLD` | НДЗ (без 2x), новые из импорта | звонить регулярно |
| `WARM` | Проинформирован, Только отправить инфо, Уже был | редкие касания (раз в 2-3 мес) |
| `HOT` | Запись на БТ, В работе | приоритет звонков |
| `CONVERTED` | Есть встреча/сделка | вне обзвонной воронки |
| `BLACKLIST` | НЕ брокер, Некорректный, Отказ, 2 НДЗ | не звоним |

### 2.3 Кампании — отдельная сущность

Сейчас в Google две колонки G и H = две разные кампании («общий обзвон» + «по Зорге»). В будущем будет больше:
- «Серебряный Бор зима 2026»
- «Новые условия комиссии май 2026»
- «БТ август 2026»

Поэтому `Campaign` как отдельная модель + `CallLog.campaignId` — обязательно.

### 2.4 Сегментация: специализация и регион

В базе нет этих полей. **Рекомендуем добавить через опросник оператора**:
- Первый успешный контакт → оператор спрашивает «коммерция или жилая?», «Москва или другой регион?»
- Поля: `specialization` (COMM / RESIDENTIAL / BOTH), `region` (MSK / SPB / OTHER)
- Через 3 месяца — данные для конверсионной аналитики

### 2.5 Расписание операторов

Делаем **позже, когда уйдём от Morekit**. Сейчас Morekit распределяет автоматически — нам ничего не делать.

Когда придёт время:
- Модель `OperatorShift` с временами начала/конца
- Round-robin + квота по дню
- Switch `LEAD_DISTRIBUTION_MODE` (MOREKIT vs OWN)

### 2.6 Связь с amoCRM

**Не дублируем amoCRM, не отказываемся от неё.** Сосуществование:
- amoCRM остаётся системой записи **лидов клиентов** (когда брокер привёл клиента)
- Наш кабинет = **админка для управления брокерами + база КЦ**
- Связь: `Broker.amoContactId` (уже есть)

Двусторонний sync **только для конкретных полей**:
- `Broker.fullName` ↔ amoCRM contact name
- `Broker.lastCallResult` ↔ amoCRM lead custom field (опционально)
- НЕ синхронизируем категорию (она наша внутренняя)

### 2.7 Импорт из Google Sheets — одноразовый

Сейчас Google = master. После импорта **переключаемся: наш кабинет = master**, Google архивируется.

Двусторонний sync с Google Sheets не делать — слишком хрупко (квоты, конфликты, лимиты записи). Если КЦ хочет видеть в Google — экспортируем raz в неделю в read-only viewer-таблицу.

### 2.8 Удаление PII при увольнении

Когда увольняется оператор/брокер — **анонимизация**, не удаление:
- Заменяем имя на «Бывший брокер #N»
- Сохраняем CallLog (для аналитики)
- Помечаем `isActive = false`, `anonymizedAt`

## 3. Открытые вопросы для обсуждения

| # | Вопрос | Мой вариант ответа |
|---|---|---|
| 1 | Хранить телефоны в открытом виде или маскировать (последние 4 цифры) для обычных операторов? | **В открытом виде** — иначе нельзя звонить. Маскирование только для админ-аналитики |
| 2 | Гражданин ЕС/Индии/etc — если оператор работает удалённо из другой страны, попадает ли база под GDPR/152-ФЗ? | Обсудить с юристом, но **в пределах РФ всё ОК если есть согласие** |
| 3 | Что если один телефон у двух брокеров (поменяли симку)? | По phone уникальный constraint, ручной merge через админку |
| 4 | Хранить запись звонка (audio из Mango)? | **Да, но в S3 с тегом TTL 6 месяцев** — для качества и юр.вопросов |
| 5 | Кто такие «Координаторы» (Sheet2 у тебя в Excel)? | Жду уточнения. Возможно — старший оператор / руководитель команды |
| 6 | Что делать с дубликатами в Google (одинаковые телефоны)? | Merge при импорте — суммировать счётчики, оставлять последний коммент |
| 7 | Сколько лет хранить «BLACKLIST» брокеров (НЕ брокер / некорректный номер)? | **2 года** — потом удалять полностью |

## 4. План реализации (по этапам)

### Этап 0: Подготовка (1 день)
- Согласовать ТЗ с тобой (это документ)
- Получить тестовый доступ к Google Sheets через Service Account
- Договориться: один ли это `Broker` или новая модель

### Этап 1: Модель данных (1 день)
- Prisma migration: добавить поля в `Broker`, создать `CallLog`, `Campaign`
- Без UI — только схема + миграция
- Деплой → автоматическая миграция через `prisma db push`

### Этап 2: Импорт из Google Sheets (2 дня)
- Скрипт `scripts/import-google-brokers.js`
- Workflow task: `task=import-brokers-from-google`
- Дедупликация по телефону (нормализация в `+7XXXXXXXXXX`)
- Создание `CallLog` записей из колонок G и H
- Установка `BrokerCategory` автоматически по результату

**Контрольная точка**: пользователь проверяет — все 10K брокеров в базе с правильными категориями? Если нет — корректировки.

### Этап 3: UI колл-центра (5-7 дней)
- Страница `/admin/call-center` — очередь обзвона + dashboard
- Карточка брокера для оператора с быстрыми кнопками
- История CallLog
- Простой роутинг между операторами (пока без расписания)

### Этап 4: Кампании (3 дня)
- CRUD кампаний
- Фильтры таргетинга
- Dashboard по каждой кампании
- Экспорт CSV

### Этап 5: Аналитика (3 дня)
- Воронка конверсии (звонок → БТ → встреча → сделка)
- KPI по операторам
- Графики

### Этап 6 (опционально): Round-robin распределение (5 дней)
- Модель `OperatorShift`
- API для управления расписанием
- Алгоритм распределения
- Switch `LEAD_DISTRIBUTION_MODE`

### Этап 7 (опционально): Mango Office интеграция (3 дня)
- Авто-логирование звонков (уже есть `mango.adapter.ts`)
- Click-to-call из карточки брокера
- Запись звонков → S3

## 5. Что НЕ делаем

- **Не отказываемся от amoCRM** — сохраняем как канал лидов клиентов
- **Не делаем двусторонний sync с Google** — only one-time import
- **Не хранится audio звонков по умолчанию** — только если решим что нужно для юр.вопросов
- **Не строим систему обучения операторов** — это другая задача
- **Не делаем мобильное приложение для оператора** — web-кабинет адаптивный должен хватить
