# Пояснительная записка: интеграция кабинета брокера ↔ amoCRM

**Дата:** 2026-06-03
**Версия:** 1.0
**Что описывает:** Как сейчас работает связка между кабинетом брокера (`broker.stmichael.ru`) и amoCRM (`stmichael.amocrm.ru`). Алгоритм фиксации клиента, проверка уникальности, дедупликация лидов и контактов.

---

## 1. Общая схема

```
[Брокер]
   ↓ /fixation (форма фиксации клиента)
[Кабинет (NextJS+Nest API)]
   ↓ POST /clients/fix
[Backend client-fixation.service.ts]
   ├── Проверка локальной БД (PostgreSQL): не фиксировал ли уже этот брокер
   ├── ⚠ Проверка уникальности в amoCRM (4 правила)
   ├── Запись в БД (Client)
   └── Создание/обновление лида в amoCRM
[amoCRM]
   ├── Контакт клиента (с custom_fields)
   ├── Контакт брокера (с IS_BROKER=true)
   ├── Лид в воронке КЦ (7600542)
   ├── Note с дублированием заявки
   └── Task «Связаться по сделке брокера»
```

## 2. Что отправляется в amoCRM при фиксации

### 2.1. Контакт клиента

При фиксации сначала ищем контакт по телефону через `findContactByPhone()`:

- **Поиск по последним 10 цифрам** телефона (`+79999999999` → `9999999999`)
- Постфильтр по `custom_fields_values.PHONE` — отсекает ложные совпадения по имени/email/комменту
- Если несколько контактов — берётся самый свежий по `updated_at`

**Если контакт найден** — обновляем его поля:
- `EMAIL` (557905)
- `REGION` (589265)
- `PRESENTATION_SENT` (835955)

**Если не найден** — создаём новый с минимальным набором полей.

### 2.2. Контакт брокера

Уже существует в amo (синхронизируется при регистрации брокера в кабинете):

- ФИО, телефон, email
- `IS_BROKER` (835415) = `true` — ключевой маркер
- `INN` агентства, `AGENCY_NAME`
- `POSITION`, `TELEGRAM_USERNAME/ID`, `WHATSAPP_USERNAME`
- `REGION`, `BROKER_TOUR_VISITED/DATE`, `PRESENTATION_SENT`

### 2.3. Лид (сделка) в воронке КЦ

Создаётся в воронке **«Воронка КЦ»** (`pipeline_id = 7600542`).

**Стандартные поля лида:**
- `name` = `Фиксация: ИВАНОВ И.И. (ZORGE9)`
- `pipeline_id = 7600542`
- `contacts` = `[клиент, брокер]` — оба прикрепляются к лиду
- `price` = сумма из «Бюджет покупки»

**Custom-поля лида (после Salesbot):**
| Field | ID | Что |
|---|---|---|
| `587387` | Тип объекта | Квартира / Апартаменты / Коммерческая |
| `583447` | Кол-во комнат | Студия / 1к / 2к / 3к / 4к+ |
| `833045` | Бюджет покупки | число в рублях |
| `604555` | Метраж м² | число |
| `583155` | Цель покупки | Себе (по умолчанию) |
| `839179` | Объект интереса | Зорге 9 / Берзарина 37 |

## 3. Проверка уникальности

Алгоритм запускается **ДО** создания/прикрепления лида. Источник истины — `amoCRM`, не наша локальная БД.

### 3.1. Шаги проверки

```
fixClient(брокер, phone, ...):
  → amoCrmAdapter.checkUniqueness(phone)
  
checkUniqueness(phone):
  1. findContactByPhone(phone) → contact
  2. if contact == null:
       return UNIQUE  // Правило 1
  3. getLeadsByContact(contact.id) → leads[]
  4. if leads.length == 0:
       return UNIQUE  // Правило 2 (нет активной истории)
  5. Для каждого лида:
       a. Получаем все его привязанные контакты
       b. Проверяем — есть ли среди них контакт с IS_BROKER=true
  6. evaluateUniqueness(leads) → verdict UNIQUE | ALARM
```

### 3.2. Четыре правила уникальности

**Правило 1 — Контакт новый:**
- В amo контакта с этим номером нет
- Действие: **UNIQUE** → создаём новый лид

**Правило 2 — Закрытые сделки:**
- Все лиды контакта в финальных статусах (`143` везде или `142` в КЦ)
- Это означает: предыдущие циклы продаж завершены
- Действие: **UNIQUE** → создаём новый лид

**Правило 3 — Активный лид «Квалифицировали, выводим на встречу» с брокером:**
- Лид в статусе:
  - `62907282` (КЦ)
  - `62907138` (Берзарина)
  - `62907154` (Зорге 9)
  - `62907170` (Толбухина)
- И есть привязанный контакт с `IS_BROKER=true`
- Действие: **UNIQUE** → прикрепляем нашего брокера к существующему лиду (НЕ создаём дубль)

**Правило 4 — Активный лид «Новое обращение / Новый Лид» с брокером:**
- Лид в статусе:
  - `62907118` (КЦ — «Новое обращение»)
  - `62907134` (Берзарина — «Новый Лид»)
  - `62907150` (Зорге 9 — «Новый Лид»)
  - `62907166` (Толбухина — «Новый Лид»)
- И есть привязанный контакт с `IS_BROKER=true`
- Действие: **UNIQUE** → прикрепляем нашего брокера к существующему

**Все остальные случаи (Встреча назначена, Сделка, Контроль оплаты и т.д.) — ALARM.**

### 3.3. Логика конкуренции брокеров

> Реальная уникальность блокирует других брокеров **только после встречи в офисе и подписанного акта осмотра**. До этого момента — это «аукцион» — несколько брокеров могут одновременно быть условно-уникальными на одном клиенте.

В коде это означает:
- На стадиях «Новое обращение» и «Квалифицировали» — все брокеры **прикрепляются к одному лиду** как разные контакты
- В amo менеджер видит «Иванов, Петров, Сидоров — все на этом клиенте»
- Кто первый доведёт до встречи и подпишет акт → `Client.fixationStatus = FIXED`
- Остальные автоматически становятся «Аннулировано» через scheduler

## 4. Как избегаем дубль карточки

### 4.1. Дубль контакта

**Раньше (баг до 2026-06-02):** `findContactByPhone` искал по `query=+79999999999` (с плюсом). Если в amo телефон сохранён как `8 (999) 999-99-99` — поиск не находил, создавался новый контакт. Получали два контакта на один номер.

**Сейчас:** Поиск по **last-10-digits** + постфильтр по `custom_fields_values.PHONE`:
1. Отправляем `query=9999999999` (только 10 цифр — формат не важен)
2. Из всех совпадений отбираем те, у которых в `custom_fields_values[PHONE].values[].value` совпадают последние 10 цифр
3. Если совпадений несколько — берём самый свежий по `updated_at`

### 4.2. Дубль лида

**Раньше:** При любой фиксации (даже на существующий контакт с активным лидом) создавался **новый лид**. На одного клиента болталось 2-3 лида от разных брокеров.

**Сейчас:** `checkUniqueness` возвращает `reusableLeadId` — ID существующего активного лида в КЦ. Если задан:
- `createFixationRequest` **не создаёт новый лид**
- Привязывает нашего брокера к существующему лиду через `POST /leads/{id}/link` (`to_entity_type=contacts`)
- Кладёт **note** в ленту лида с пометкой `🟢 Аукция уникальности — новый брокер на этом клиенте`
- Создаёт **задачу** «Связаться по сделке брокера» с типом «Аларм»

## 5. Формат заметок и задач в amo

### 5.1. Заметка (note)

**Содержание** (длинная — дублирование всей заявки):

```
📝 Фиксация клиента от брокера
Клиент: ИВАНОВ ИВАН ИВАНОВИЧ
Телефон: +79999999999
Email: ivanov@example.ru
Регион: Москва

Проект: Зорге 9
Тип: Апартаменты
Комнат: 2
Метраж: 80 м²
Бюджет: 25 000 000 ₽
Планирует покупку: от 1 до 3 месяцев
Готовность к сделке: Горячий

Брокер-агент: +79255724180
Агентство: ИП Петров (ИНН 7123456789)

Комментарий брокера: Клиент готов к встрече на этой неделе
```

**Заголовок меняется по сценарию:**
- `📝 Фиксация клиента от брокера` — новая фиксация (новый лид)
- `🟢 Аукция уникальности — новый брокер на этом клиенте` — прикрепление к существующему
- `⚠️ АЛАРМ уникальности — требуется ручная проверка КЦ` — ALARM

### 5.2. Задача (task)

**Параметры:**
- **Название**: `Связаться по сделке брокера — клиент ИВАНОВ И.И. (+79999999999), брокер +79255724180`
- **Тип**: «Аларм» (через ENV `AMO_ALARM_TASK_TYPE_ID`, fallback на 1=Звонок)
- **Срок**: +30 минут от создания
- **Распределение**: НЕ указываем `responsible_user_id` → Морикит распределяет на свободного КЦ-менеджера

## 6. Статусы в кабинете брокера ↔ статусы в amo

### 6.1. Локальные статусы клиента

| Статус | Когда выставляется | Что видит брокер |
|---|---|---|
| `CONDITIONALLY_UNIQUE` | Verdict UNIQUE, новая запись | «Уникален» |
| `UNDER_REVIEW` | Verdict ALARM | «На проверке» |
| `REJECTED` | Менеджер КЦ отклонил | «Отклонён» |
| `EXPIRED` | Истёк срок уникальности (30 дней) | «Истёк» |

### 6.2. Локальные статусы фиксации

| Статус | Когда выставляется |
|---|---|
| `NOT_FIXED` | По умолчанию (после фиксации) |
| `FIXED` | Подписан акт осмотра на встрече |
| `EXPIRED` | Срок прошёл, акт не подписан |
| `ANNULLED` | Сделка ушла к другому брокеру (после FIXED одного из конкурентов) |

### 6.3. Как меняются статусы — план на основе уточнений пользователя

> Пока в коде только начальные статусы (`CONDITIONALLY_UNIQUE`, `UNDER_REVIEW`). Остальные переходы — в работе (Фаза 3).

**После Фазы 3** (webhook + cron):

| Событие в amo | → Статус в кабинете |
|---|---|
| Менеджер КЦ прикрепил брокера к лиду | `CONDITIONALLY_UNIQUE` (одобрено) |
| Лид закрыт (`status=143`) | `REJECTED` |
| Прошло 5 рабочих часов (9:00-21:00) без прикрепления | `REJECTED` |
| Подписан акт осмотра (`Client.inspectionActSigned=true`) | `fixationStatus=FIXED` |

## 7. Дополнительные моменты

### 7.1. Обработка ошибок amo

Если amo упал во время фиксации (401/таймаут/5xx):
- Клиент **всё равно создаётся** в нашей БД (фиксация состоялась)
- В `Client.amoSyncStatus` пишется `FAILED` + `amoSyncError`
- Запись в audit (`AMO_SYNC_FAILED`)
- Уведомление координаторам и менеджерам
- Брокер видит сообщение «Клиент зафиксирован в кабинете, передача в amoCRM повторится автоматически»
- Scheduler ретраит синхронизацию каждые 30 минут

### 7.2. Обработка повторной фиксации (тот же брокер)

Если брокер пытается зафиксировать клиента, которого УЖЕ фиксировал:
- Сценарий 1: предыдущая запись с `CANCELLED` сделкой → переоткрываем лид в amo (`reopenLead`)
- Сценарий 2: предыдущая запись в `PENDING/SIGNED` сделке → ALARM, конфликт
- Сценарий 3: предыдущая запись активна без сделок → требуем подтверждение `confirmDuplicate=true`

### 7.3. Cross-broker конфликт (локальный)

Если в нашей БД уже есть `Client` с этим телефоном у **другого брокера** в статусе:
- `CONDITIONALLY_UNIQUE` (не истёк)
- `FIXED`
- С проведённой встречей (`COMPLETED/CONFIRMED`)

→ Создаём в нашей БД запись `UNDER_REVIEW` + note в amo лид «попытка повторной фиксации».

## 9. Что в работе (TODO)

**Фаза 3 — синхронизация статусов из amo обратно в кабинет:**

1. **Webhook от amo** `POST /api/webhooks/amo/lead-updated`:
   - При прикреплении/отвязке брокера от лида → пересчёт `Client.uniquenessStatus`

   - Сейчас `responsible_user_id` не задаём → надеемся на Морикит
